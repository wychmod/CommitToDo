# CommitToDo 用户页面与功能规划

## 1. 规划目标

本规划只讨论用户进入应用后实际使用的页面。

官网首页已经确定，不再纳入应用页面规划。首页中的“进入工作台”
进入应用工作区，“查看任务流”用于展示仓库、分支、任务和提交之间的关系。

应用页面围绕以下用户行为组织：

1. 查看今天应该做什么。
2. 进入一个仓库，持续推进目标。
3. 查看任务的历史、分支关系和长期节奏。
4. 搜索内容或快速执行操作。
5. 管理数据、提醒和应用偏好。

产品核心关系保持为：

`工作区 → 仓库 → 分支 → 任务 → 提交`

## 2. 最终页面结构

建议应用保留 **5 个用户主页面**：

| 页面 | 用户要解决的问题 | 是否独立页面 |
|---|---|---|
| 今日工作台 | 我今天要做什么，最近在推进什么？ | 是 |
| 仓库概览 | 这个目标当前整体进展如何？ | 是 |
| 仓库任务 | 当前分支有哪些任务，下一步做什么？ | 是 |
| 洞察 | 我完成了什么，分支如何演进，节奏如何？ | 是 |
| 设置 | 如何管理外观、提醒、本地数据和仓库？ | 是 |

以下功能不作为独立桌面页面：

- 搜索：桌面端使用全局搜索/命令面板。
- 任务详情：桌面端使用右侧详情面板。
- 新建和编辑任务：桌面端使用右侧表单面板。
- 新建、合并和删除分支：使用弹窗或菜单。
- 通知、账户、本地保存状态：使用顶部入口、菜单或状态提示。

移动端由于屏幕空间有限，搜索、任务详情和任务表单可以表现为全屏页面，
但它们仍然属于对应主页面的操作流程，不增加新的一级导航。

## 3. 页面一：今日工作台

### 用户目的

用户打开应用后快速确认今天应该推进哪些任务，并进入最近使用的仓库。

### 页面内容

- 今日任务。
- 逾期任务。
- 已完成但尚未形成最终提交的任务。
- 最近使用的仓库。
- 本周完成数量和连续完成天数。
- 最近活动摘要。
- 本地保存和离线状态。

### 主要操作

- 完成或恢复任务。
- 快速新建任务。
- 新建仓库。
- 进入仓库。
- 打开任务详情。

### 合并现有功能

将当前工作台中的以下区域重新组织到此页面：

- 工作台统计卡。
- 今日脉冲。
- 最近仓库。
- 快捷操作。
- 本地状态。

统计信息只保留能够帮助用户决定下一步行动的内容，避免多处重复展示相同数字。

### Web 布局

- 左侧：应用一级导航和最近仓库。
- 中间：今日任务，按仓库分组。
- 右侧：本周节奏、最近仓库和快速添加。

### 手机布局

- 第一屏优先显示今日任务。
- 仓库名称作为任务分组标题。
- 本周节奏和最近仓库放在任务列表之后。
- 底部导航中作为“今日”入口。

---

## 4. 页面二：仓库概览

### 用户目的

用户进入一个仓库后，快速理解当前有哪些分支、正在推进哪些任务、
最近发生了哪些提交以及整体完成节奏。

### 页面内容

页面结构参考已经确定的首页软件预览：

- 仓库信息和仓库切换。
- 分支列表及每条分支的任务数量。
- 各分支的重点任务摘要。
- 最近提交时间线。
- 仓库热力图摘要。
- 完成任务数、连续完成天数、本月提交数。

### 主要操作

- 切换仓库。
- 切换分支。
- 进入完整任务页。
- 新建任务。
- 新建分支。
- 查看全部提交。
- 查看完整热力图或 Graph。

### 页面边界

概览用于理解仓库状态，不在这里承担复杂筛选和批量任务管理。

首页图片中的四栏可以作为概览页的主要结构：

`仓库导航 / 分支 / 任务摘要 / 提交与热力图`

### Web 布局

- 左栏：仓库内导航。
- 第二栏：分支列表。
- 中间主区域：按分支分组的任务摘要。
- 右侧：最近提交与热力图摘要。

### 手机布局

- 顶部显示仓库和当前分支。
- 优先显示任务摘要。
- 分支列表通过抽屉打开。
- 提交和节奏信息放在任务摘要之后。

---

## 5. 页面三：仓库任务

### 用户目的

用户在当前仓库和分支中创建、筛选、更新和完成任务。

### 页面内容

- 分支树。
- 当前分支信息。
- 任务列表。
- 状态筛选：全部、今天、待办、进行中、已完成、已取消。
- 优先级和截止日期筛选。
- 任务排序。
- 任务详情面板。
- 任务相关提交记录。

### 主要操作

- 新建和编辑任务。
- 完成任务并生成提交。
- 恢复已完成任务。
- 删除任务。
- 切换分支。
- 新建、合并、归档或删除分支。

### 与仓库概览的关系

- 仓库概览回答“这个仓库现在怎么样”。
- 仓库任务回答“我现在具体要做什么”。

两者共享仓库和分支上下文，但不应堆在同一个超长页面中。

### 任务详情与表单

桌面端：

- 点击任务后，在右侧打开详情面板。
- 新建和编辑任务复用同一右侧表单面板。
- 完成操作明确命名为“完成并提交”。

手机端：

- 任务详情使用全屏页面。
- 新建和编辑使用全屏表单或底部面板。
- 返回任务列表时保留原分支、筛选和滚动位置。

---

## 6. 页面四：洞察

### 用户目的

用户回顾已完成的工作、理解分支演进，并观察自己的长期行动节奏。

### 合并的现有页面

以下页面合并到一个洞察页面：

- 提交记录。
- 仓库 Git Graph。
- 全局 Git Graph。
- 仓库热力图。
- 全局热力图。

### 页面结构

顶部提供范围选择：

- 全部仓库。
- 指定仓库。
- 指定分支。

主体提供三个标签：

#### 活动

- 按日期展示提交记录。
- 显示提交类型、任务、仓库、分支和时间。
- 支持按仓库、分支和时间筛选。

#### 图谱

- 展示分支创建、提交和合并关系。
- 点击节点查看对应提交和任务。
- 支持缩放、拖动和重置视图。

#### 节奏

- 展示热力图。
- 展示完成任务数、提交数、连续完成天数和单日最高值。
- 支持年度和月份范围切换。

### 为什么合并

提交记录、Graph 和热力图本质上都是对同一批提交数据的不同观察方式。
用户应先选择观察范围，再切换表达方式，而不是在多个相似页面之间跳转。

### 手机布局

- 顶部固定范围选择和三个标签。
- 活动使用纵向时间线。
- Graph 支持双指缩放和拖动。
- 热力图允许横向滚动。

---

## 7. 页面五：设置

### 用户目的

用户管理应用外观、任务提醒、本地数据、仓库属性和安装状态。

### 页面分区

#### 外观

- 深色、浅色或跟随系统。
- 强调色。
- 界面密度。

#### 提醒

- 是否开启截止提醒。
- 默认提前时间。
- 通知权限状态。

#### 数据与备份

- 本地存储状态。
- 导出 JSON、CSV 和 Markdown。
- 导入 JSON。
- 清理本地数据。

#### 仓库设置

- 仓库名称和描述。
- 默认分支。
- 归档或删除仓库。

#### 应用与账户

- PWA 安装状态。
- 登录或账户入口。
- 数据同步能力在实现后再开放，不提前展示不可用功能。

### 全局设置与仓库设置的合并方式

保持一个设置页面，通过左侧分区或顶部范围明确区分：

- 应用设置。
- 当前工作区设置。
- 当前仓库设置。

不再保留单独的仓库设置页面。

## 8. 全局搜索与命令面板

### 定位

搜索是全局能力，不是桌面端一级页面。

### 搜索范围

- 仓库。
- 分支。
- 任务。
- 提交。

### 快捷命令

- 新建仓库。
- 新建任务。
- 新建分支。
- 切换仓库或分支。
- 打开今日工作台、仓库、洞察或设置。

### 平台表现

- Web：通过顶部搜索框或 `Ctrl/Command + K` 打开浮层。
- 手机：底部导航提供搜索入口，打开全屏搜索。

当前全局搜索页和仓库搜索页合并到此能力。

---

## 9. 最终导航

### Web 一级导航

1. 今日。
2. 仓库。
3. 洞察。
4. 设置。

顶部常驻：

- 工作区/仓库切换。
- 全局搜索。
- 新建菜单。
- 本地保存或同步状态。
- 通知。
- 账户。

### 仓库内导航

1. 概览。
2. 任务。
3. 洞察。
4. 设置。

“分支”不单独成为页面，它是概览和任务页中的核心上下文。

“提交”与“热力图”不单独成为页面，它们进入洞察。

### 手机底部导航

1. 今日。
2. 仓库。
3. 洞察。
4. 搜索。

设置从账户菜单进入，避免底部导航超过四个主要入口。

## 10. 推荐路由

| 路由 | 页面/能力 |
|---|---|
| `/workspace` | 今日工作台 |
| `/repository/:id` | 仓库概览 |
| `/repository/:id/tasks` | 仓库任务 |
| `/insights` | 洞察，通过参数选择全部仓库或指定仓库 |
| `/settings` | 设置，通过参数选择应用、工作区或仓库范围 |
| `/search` | 仅用于手机全屏搜索和可分享的搜索链接 |

上下文路由：

| 路由 | 表现 |
|---|---|
| `/repository/:id/task/new` | Web 打开任务表单面板，手机打开全屏表单 |
| `/repository/:id/task/:taskId` | Web 打开任务详情面板，手机打开全屏详情 |
| `/repository/:id/task/:taskId/edit` | Web 打开编辑面板，手机打开全屏表单 |

洞察参数示例：

- `/insights?scope=all&tab=activity`
- `/insights?repository=:id&tab=graph`
- `/insights?repository=:id&branch=:branchId&tab=heatmap`

## 12. 必须覆盖的用户状态

这些不是独立页面，但每个主页面都必须处理相关状态：

- 首次进入应用。
- 没有仓库。
- 仓库没有任务。
- 分支没有任务。
- 无搜索结果。
- 加载失败和重试。
- 离线使用。
- 本地保存成功或失败。
- 任务完成和撤销。
- 分支合并确认和失败。
- 导入导出成功或失败。
- PWA 可安装、已安装或不支持。

## 13. 最终结论

应用不需要继续维护当前数量的独立页面。

最终应围绕 **今日工作台、仓库概览、仓库任务、洞察、设置**
这 5 个用户主页面建设。

搜索、任务详情、任务表单和分支操作属于跨页面能力或上下文操作；
提交记录、Graph 和热力图统一属于洞察。

这样既保留了当前项目已经实现的全部核心能力，也让用户始终能够回答：

- 我今天做什么？
- 这个目标推进到哪里了？
- 当前分支下一步做什么？
- 最近完成了什么？
- 我的长期节奏如何？
