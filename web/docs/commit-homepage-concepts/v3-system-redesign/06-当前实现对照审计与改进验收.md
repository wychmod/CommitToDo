## 1. 给执行模型的任务定义

目标不是机械复刻截图中的数据，而是让现有 5 个应用页达到参考图的视觉完成度、
信息层级和操作效率，并补齐真实使用中的关键状态。

必须遵守以下判断边界：

1. 截图里的导航项、分支、仓库、任务、提交数量与当前数据不同，不是问题。
2. 同一组件因真实数据多少而高度不同，不直接判定为样式问题。
3. 需要对齐的是布局比例、视觉层级、对比度、控件形态、状态表达和工作流。
4. 不能为了像截图而伪造业务数据、写死列表或破坏真实数据流。
5. 首页和落地页是只读参考，不得修改
   `src/presentation/screens/landing/**`。
6. 不得把 V3 样式全局导入 `src/main.tsx` 或 `src/app.tsx`。
7. 不得用裸色值替代 `--v3-*` Token，也不得在同一页面混用
   `--v3-*` 与 `--color-*`。
8. 现有工作区有未提交改动。实施前必须先读 `git diff`，只在当前改动上增量工作，
   不得回退用户改动。

### 1.1 完成定义

只有同时满足以下条件，任务才算完成：

- 5 个页面在参考图对应桌面尺寸下具有一致的视觉骨架和清晰层级。
- 5 个页面的核心交互都经过真实浏览器验证，而不是只通过单元测试。
- 设置页不存在重复加载或持续重渲染，主题选择在 V3 页面实际生效。
- 仓库任务页的优先级多选是可理解、可键盘操作的弹出式多选控件。
- 手机和窄屏不出现固定 252px 侧栏挤压、横向溢出或不可达操作。
- 空数据、加载、错误、离线、保存中和成功反馈都具有明确的下一步动作。
- `npm run lint`、`npx tsc -b`、`npm test`、`npm run build` 全部通过。

## 2. 设计规格

### 2.1 Purpose Statement

CommitToDo 面向需要把仓库、分支、任务和提交持续关联起来的个人用户。
应用页必须帮助用户在一眼内确认“现在做什么、做到哪里、下一步是什么”，同时保留
local-first 工具的可信感。界面应像一套专注工作的 Git 仪表台，而不是营销网站或
通用卡片后台。

### 2.2 Aesthetic Direction

**工业化、工具型、克制但高辨识度的暗色工作台。**

视觉吸引力来自精确对齐、稳定的多栏结构、明暗层级和少量高亮，不来自大面积渐变、
发光装饰或夸张动画。荧光绿色只用于主行动、当前上下文、成功状态和关键数据；青色、
蓝色、橙色、红色只承担分支或语义状态。

### 2.3 Color Palette

项目已有 V3 品牌系统，以下为强制覆盖，不引入新的页面私有色板：

| 角色 | Token | 当前值 | 使用原则 |
| --- | --- | --- | --- |
| 主画布 | `--v3-bg` | `#000000` | 最大面积背景 |
| 卡片 | `--v3-card` | `#0c0f0d` | 单个信息容器 |
| 控件 | `--v3-control` | `#141716` | 输入、切换和次级悬停 |
| 强文本 | `--v3-text-strong` | `#f5f5f2` | 标题、主数据、关键动作 |
| 主强调 | `--v3-primary` | `#80e48c` | 当前项、主按钮、成功 |
| 分隔边框 | `--v3-border` | `#30332f` | 可交互容器边界 |

参考图中的主色看起来更偏黄绿，但不得直接在页面中写一个新的 lime hex。
先通过面积、对比和层级恢复参考图的吸引力。只有在产品明确批准全局品牌色变更时，
才允许调整 `--v3-primary`，且必须按项目规范同步 Token 文档和参考实现。

### 2.4 Typography

- 中文和正文沿用项目批准的 `--v3-font-sans` / Tailwind `font-sans`。
- 分支名、提交哈希、日期短码使用 `--v3-font-mono` / `font-mono`。
- 页面标题 40-54px；区块标题 18-20px；主正文 14-16px；辅助信息 12-13px。
- 不允许为了“更科技”而把大段正文改成等宽字体。
- 标题与数据使用字重建立层级，正文不要普遍低于当前 `--v3-text-secondary`。

### 2.5 Layout Strategy

- 桌面使用固定应用壳层：顶部命令栏、左导航、底部状态栏和页面工作区。
- 内容区采用不对称多栏：主工作区明显宽于上下文栏，右栏保持稳定宽度。
- 主数据区尽量无装饰；卡片只包裹真正独立的模块，不嵌套卡片。
- 桌面首屏要呈现任务或核心图形，而不是大面积标题、空白或说明文案。
- 窄屏必须切换为移动导航和单栏内容，不能继续使用固定左边距。

### 3.2 当前实现证据

审计使用当前工作区、Vite 开发服务器和真实浏览器完成。今日工作台、仓库概览、
仓库任务、洞察均能识别正确路由、标题和主要 DOM，未观察到框架错误覆盖层或相关
控制台错误。

设置页在真实浏览器读取过程中持续重渲染并拖住页面检查。代码证据是：

```tsx
const store = useSettingsScreenStore();

React.useEffect(() => {
  void store.load();
}, [store]);
```

Zustand 返回的整个 store 对象会在状态更新后变化，`load()` 又会更新状态，因此该
effect 可能反复执行。现有组件测试把 store mock 为稳定对象，无法覆盖真实订阅行为。

### 3.3 不应误判的差异

- 当前本地数据中存在同名仓库、不同任务总数或没有提交，这些不作为视觉缺陷。
- 参考图展示 3 个分支而当前仓库可能只有 1 个，不要求补假分支。
- 洞察图中节点多少由真实提交决定，不以节点数量做像素对齐。
- 设置页 PWA 按钮是否可用由浏览器能力决定，不强制显示参考图的“可安装”状态。

## 4. 全局对照结论

### 4.1 当前实现已经做对的部分

- 5 个页面已经统一使用 `V3AppShell`，整体信息架构与参考图一致。
- 顶部命令栏、左导航、底部状态栏和 local-first 状态已经建立。
- 今日页的中间任务流与右侧节奏栏、概览页三栏、任务页四区、洞察主区加右栏、
  设置主区加右栏，骨架方向都正确。
- 主按钮、活动导航、任务选中态、分支颜色和语义色已使用 V3 Token。
- 任务详情、表单、空状态和确认弹窗已有基础实现，不需要推倒重写。

### 4.2 主要差距

| 优先级 | 差距 | 当前表现 | 目标 |
| --- | --- | --- | --- |
| P0 | 设置页重复加载 | 真实浏览器检查被持续渲染拖住 | 首次进入只加载一次，状态更新不触发重新 load |
| P0 | 壳层不响应窄屏 | 固定 `w-[252px]` 与 `ml-[252px]` | 窄屏切换移动导航，内容占满可用宽度 |
| P1 | 优先级筛选控件错误 | 39px 高的原生 `select multiple`，选项语义和显示都差 | 按钮 + 多选菜单 + 已选摘要 + 清除 |
| P1 | V3 浅色模式不完整 | `v3-tokens.css` 只有暗色 `:root` 值 | 设置为浅色后 5 页画布和控件真实切换 |
| P1 | 整体视觉重量偏轻 | 边界、次级文本、卡片层级在当前截图中近乎融入黑底 | 参考图那样一眼分清导航、工作区、上下文栏 |
| P1 | 空状态占据过大工作区 | 大块居中说明，缺少视觉结构和上下文 | 保留模块轮廓、说明原因并给唯一主动作 |
| P1 | 首屏密度不稳定 | 有数据时尚可，少数据时出现大面积无意义黑区 | 版面仍保持完整且有明确下一步 |
| P2 | 字号和控件节奏偏小 | 顶栏品牌和辅助动作存在 13px 密集感 | 主路径 14-16px，辅助信息才使用 12-13px |
| P2 | 状态反馈分散 | 保存、离线、错误反馈样式和位置不完全统一 | 状态栏负责长期状态，Toast/行内提示负责事件 |

### 4.3 全局改进要求

#### G1. 修复应用壳层响应式行为（P0）

涉及：

- `src/presentation/components/v3/v3-app-shell.tsx`
- `src/presentation/components/v3/v3-left-navigation.tsx`
- `src/presentation/components/v3/v3-top-command-bar.tsx`
- `src/presentation/components/v3/v3-bottom-status-bar.tsx`

要求：

- `>= 1280px`：保留完整 252px 左导航和页面右栏。
- `1024-1279px`：允许压缩或抽屉化最近仓库区，主导航仍可达。
- `< 1024px`：左导航不再常驻，主内容不得保留 `ml-[252px]`。
- `< 768px`：顶部只保留品牌、搜索/命令入口和新建入口；次要状态移入菜单。
- 移动端提供底部一级导航：今日、仓库、洞察、搜索；设置从用户菜单进入。
- 底部状态栏在手机上不能与底部导航重叠。

验证：360x800、768x1024、1024x768、1536x1024 四个视口无横向滚动，
主要动作可见且可点击。

#### G2. 恢复参考图的层级和质感（P1）

要求：

- 页面背景、侧栏、卡片、控件至少形成 4 个可辨认的明度层级。
- 外层布局分隔使用 `--v3-border-soft`，可交互卡片和输入使用
  `--v3-border`；不要所有边界都用最弱线。
- 主卡片 hover 只轻微提高背景与边框，不增加大阴影或发光。
- 当前项使用“背景 + 3px 左边/底边强调 + 强文本”三重信号，不能只靠颜色。
- 主按钮高对比，次按钮保持描边；危险动作只在危险区域使用红色。
- 保持卡片圆角不超过 8px，避免卡片套卡片。

#### G3. 建立稳定的字体与间距层级（P1）

要求：

- 页面眉题 12px；H1 40-54px；区块标题 18-20px；行主文本 14-16px。
- 顶部品牌名不小于 15px，一级导航保持 15px。
- 任务/表格行高度桌面 48-54px；工具栏控件 38-40px。
- 页面标题区、内容区、右栏的顶部基线在同一水平系统内。
- 长标题、仓库名、分支名必须截断或换行，不能挤压动作区。

#### G4. 统一状态与反馈（P1）

- 加载状态使用与最终布局同尺寸的骨架或结构占位，减少跳动。
- 空状态必须包含：发生了什么、为什么、唯一主动作。
- 错误状态必须包含重试；不可只提供“清除错误”。
- 保存状态语义：`saving`、`saved`、`offline`、`error` 都要可见且带
  `aria-live`。
- 动作成功用 Toast；长期状态放在顶栏/底栏，不重复堆叠。

#### G5. 完成浅色主题（P1）

当前 `ThemeProvider` 会切换 `data-theme`，但 V3 Token 只有暗色值。
应为 V3 定义完整浅色语义值，所有 5 页仍只消费 `--v3-*`。不得改成应用壳层的
`--color-*`。

验证浅色时至少检查：画布、导航、卡片、输入、主按钮、选中行、状态 Chip、
热力图 5 档、图谱节点和焦点环。

## 5. 页面一：今日工作台 `/workspace`

### 5.1 参考图意图

首屏先回答“今天最重要的事情是什么”，再给出完成节奏和最近仓库。中间任务流是
绝对主角，右栏是决策辅助，不应与任务列表争夺注意力。

### 5.2 当前实现与参考图的差异

| 维度 | 当前实现 | 参考图 / 目标 | 判定 |
| --- | --- | --- | --- |
| 骨架 | 左导航 + 主任务区 + 388px 右栏 | 相同 | 已对齐 |
| 标题区 | 标题、日期、两个动作齐全 | 相同 | 已对齐 |
| 视觉对比 | 任务分组和行分隔偏弱，黑色面积占比过高 | 组标题、行、状态在一眼内可扫描 | 需改进 |
| 筛选 | 全部/今天/逾期/待提交 + 排序 | 相同 | 已对齐 |
| 任务行 | 字段齐全，但辅助文字和行边界偏暗 | 日期、优先级、状态形成明确列节奏 | 需改进 |
| 右栏 | 节奏、热力图、最近仓库齐全 | 相同 | 已对齐 |
| 少数据状态 | 容易出现大面积空黑 | 仍保持完整节奏与清晰 CTA | 需改进 |

不要把本地出现的同名仓库或任务数量与截图不同列为视觉问题。

### 5.3 改进方案

1. 保持标题区高度和双按钮结构，不增加营销式说明。
2. 把“今日焦点”标题、完成数和进度条组织成同一视觉组；进度条宽度随容器响应，
   不写死到窄屏溢出。
3. 筛选区在长列表滚动时保持可达。桌面可在主内容滚动容器内做轻量 sticky，
   但不能遮挡分组标题。
4. 任务分组标题与首行间距控制在 12-16px，组间距 20-24px；行高 48-52px。
5. 行内信息按“完成框 → 标题 → 日期 → 优先级 → 状态 → 负责人”稳定排列。
6. 完成任务后立即更新行、进度环和统计，并显示可撤销反馈；不要整页 loading。
7. 热力图空格与有数据格的明度差必须足够，hover/focus 显示精确日期和数量。
8. 最近仓库项整行可点击，hover 提高背景和边框，仓库色只作小面积识别。
9. 无仓库时显示“创建第一个仓库”；有仓库无任务时显示“快速添加任务”。

### 5.4 需要覆盖的状态

- 默认有任务；今天无任务；只有逾期任务；只有待提交任务。
- 完成、恢复、完成失败、离线完成。
- 无仓库；有仓库无任务；加载失败并重试。
- 长任务标题、长仓库名、10 个以上任务分组。

### 5.5 验收

- 1536x1024 首屏能看到标题、今日焦点、至少一个任务分组和右栏三个模块的起始部分。
- 点击“今天/逾期/待提交”后，按钮选中态和任务集合同步变化。
- 点击任务打开详情；关闭后筛选和滚动位置保留。
- 完成任务后统计同步，Toast 提供撤销或明确成功结果。
- 360px 宽度下任务元信息分两行，不与标题重叠；右栏移到任务列表之后。

## 6. 页面二：仓库概览 `/repository/:id`

### 6.1 参考图意图

用一屏回答“这个仓库现在推进到哪里”。左侧是分支地图，中间是当前推进，右侧是
最近提交和长期节奏。页面是仓库驾驶舱，不承担复杂筛选。

### 6.2 当前实现与参考图的差异

| 维度 | 当前实现 | 参考图 / 目标 | 判定 |
| --- | --- | --- | --- |
| 三栏比例 | 284px / 弹性主栏 / 399px | 基本一致 | 已对齐 |
| 仓库标题 | 名称、描述、最近活动、双动作齐全 | 相同 | 已对齐 |
| 空描述 | 显示“这个仓库还没有描述” | 应给出可行动的补全入口 | 需改进 |
| 分支卡 | 数据齐全，当前分支有强调 | 相同 | 已对齐 |
| 当前推进 | 结构正确，少数据时下方空白明显 | 保持模块完整但不显得未完成 | 需改进 |
| 最近提交空态 | 大块空白中央放文案 | 有结构感、有下一步 | 需改进 |
| 热力图 | 数据和统计齐全 | 相同 | 需提升对比 |

### 6.3 改进方案

1. 保持现有三栏尺寸，它与参考图已经接近，不要重做为通用卡片网格。
2. 仓库没有描述时使用低权重“添加描述”动作，编辑可复用仓库设置能力。
3. 分支概况卡的每个分支块整块可点击；当前分支同时具备左侧色条、边框和标签。
4. 当前推进按分支分组；每组头部显示分支、任务数、折叠状态。
5. 少于 3 个任务时不要把行强行拉伸；把剩余空间留给清晰的“查看全部任务”
   收口动作或状态摘要。
6. 最近提交空状态保留时间线视觉锚点或提交图标，文案下提供“去完成任务”链接。
7. 热力图使用完整 5 档，月份标签与网格基线对齐，统计值使用主色但标签保持中性。
8. 三栏卡片顶部对齐；右栏两卡间距与中栏模块间距一致。

### 6.4 需要覆盖的状态

- 1/3/10 个分支；长分支名；无默认分支。
- 当前分支有任务、无任务、只有已完成任务。
- 有提交、无提交、提交加载失败。
- 仓库无描述、已归档、仓库不存在。

### 6.5 验收

- 1672x941 下三栏比例和参考图一致，主栏不被右栏挤压。
- 点击分支卡后，分支强调与当前推进内容同步变化。

## 9. 页面五：设置 `/settings`

### 9.1 参考图意图

设置页是低频但高信任页面。主区按外观、提醒、数据、仓库分段，右栏解释
local-first、PWA 和账户状态。页面应像系统偏好设置，不应堆叠许多卡片。

### 9.2 当前实现与参考图的差异

| 维度 | 当前实现 | 参考图 / 目标 | 判定 |
| --- | --- | --- | --- |
| 范围切换 | 应用/工作区/当前仓库齐全 | 相同 | 已对齐 |
| 分段内容 | 外观、提醒、数据、仓库齐全 | 相同 | 已对齐 |
| 右栏 | local-first、PWA、账户齐全 | 相同 | 已对齐 |
| 首次加载 | 整个 store 作为 effect 依赖，可能重复 load | 稳定进入、只加载一次 | P0 缺陷 |
| 主题切换 | Store 和 `data-theme` 会变，V3 Token 不切浅色 | 选择立即真实预览 | P1 缺陷 |
| 分段节奏 | `gap-12`，首屏可能显得过松 | 参考图更像连续系统表单 | 需改进 |
| 保存反馈 | 有文字状态 | 应避免因反复 load 闪烁 | 需修复 |

### 9.3 改进方案

#### 9.3.1 先修重复加载（P0）

不要订阅整个 Zustand store。使用 selector 分别取得字段和稳定 action：

```tsx
const load = useSettingsScreenStore((state) => state.load);

React.useEffect(() => {
  void load();
}, [load]);
```

其他字段也使用 selector 或结构化 selector + `useShallow`，避免任意设置变化让整个页面
和事件订阅 effect 重跑。PWA 事件 effect 只依赖稳定 action。

必须增加真实 store 回归测试：渲染 `SettingsScreen` 后等待 load 完成，断言 load 调用一次，
再改变 scope/保存状态，断言不会再次 load。

#### 9.3.2 完成主题能力

- 深色、浅色、跟随系统切换必须立即改变当前 V3 页面。
- 强调色切换必须更新主按钮、选中态、焦点环和热力图高档，但不能破坏语义红/橙。
- “界面预览”必须消费真实 Token，不制作与实际页面脱节的静态样例。
- 系统主题变化只在 `themeMode === 'system'` 时响应。

#### 9.3.3 表单密度和分区

1. 主区保持无卡片的连续分段，每段用标题和分隔线组织。
2. 分段间距从当前 48px 评估收敛到 32-40px，使 1536x1024 首屏至少看到
   外观、提醒和数据区开头。
3. Label、Control、Status 使用稳定三列基线；输入宽度不随内容抖动。
4. 保存状态放在范围切换下方并保持固定高度，避免“保存中/已保存”引发布局移动。
5. 导入、清理、删除等高风险操作必须二次确认并明确影响范围。
6. PWA 不支持时说明原因；可安装时才显示主色安装按钮；已安装时显示稳定状态。
7. 登录未实现时保持禁用或隐藏，不展示像可用主流程的假入口。

### 9.4 需要覆盖的状态

- 深色、浅色、跟随系统；5 个强调色；舒适/紧凑密度。
- 通知默认、允许、拒绝、不支持。
- IndexedDB 正常、不可用；导入预览、冲突、失败、成功；三种导出。
- 有当前仓库、无当前仓库；保存中、成功、失败、离线。
- PWA 可安装、已安装、不支持。

### 9.5 验收

- 1536x1024 页面稳定加载，浏览器控制和 DOM 读取不被持续渲染拖住。
- 初次加载一次；切换 scope、主题、密度或编辑仓库后不重复执行 load。
- 深/浅主题切换后画布、卡片、输入和文本均改变，刷新后保持选择。
- 主题和强调色控件都有可见 focus，纯键盘可操作。
- 导入/导出/删除路径有成功和失败证据；危险操作不能一键误触。

## 14. 交付报告模板

后续模型完成实施时，最终回复必须包含：

1. **Summary**：用户可见变化和是否验收通过。
2. **Files changed**：按共享组件和 5 个页面分组。
3. **Checks**：页面身份、非空、无覆盖层、控制台、截图、交互证明。
4. **Interaction loop**：逐页写明实际点击路径和观察结果。
5. **Commands**：四个质量门禁的结果。
6. **Intentional deviations**：哪些差异来自真实数据或产品约束，而不是遗漏。
7. **Remaining risk**：仍未覆盖的浏览器、数据规模或系统能力。
8. **Screenshots**：最后集中放参考尺寸桌面图和移动图，不与正文穿插。

不接受只写“已按截图还原”“测试通过”而没有可复核证据的交付。
